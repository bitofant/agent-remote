# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

**Everything in CLAUDE.md must be extremely terse: bulleted lists, not paragraphs, as concise and token-efficient as possible.**

**Keep this file correct: update it when details change, and add genuinely important architectural decisions as they're made.**

## Project state

- Scaffolded. Working vertical slice: launch Claude Code/pi sessions, stream live to xterm in the browser, type back, switch between concurrent sessions.
- TypeScript throughout. Tests use **Vitest** (`npm test`), co-located `*.test.ts` next to source, Node environment (`vitest.config.ts`). First coverage is the pure shared logic (`shared/chat.test.ts` reducer, `shared/render.test.ts`) — no DOM, no processes, no tokens. Test files are excluded from `npm run typecheck` (both tsconfigs) so that gate stays scoped to shipping code; Vitest runs them. Component tests (jsdom + Testing Library) and live smoke scripts layer on top later.

## Commands

- `npm run dev` — single-process dev server (`tsx watch server/index.ts --dev`).
- `npm run build` — production frontend build to `dist/web` (Vite).
- `npm start` — run server serving the prebuilt `dist/web`.
- `npm run typecheck` — `tsc --noEmit` for web (`tsconfig.json`) + server (`tsconfig.server.json`).
- `npm test` — Vitest run (once, pure/fast — no processes/tokens); `npm run test:watch` for the watch loop. Excludes `*.e2e.test.ts`.
- `npm run test:e2e` — live end-to-end tests (`*.e2e.test.ts`, `vitest.e2e.config.ts`) that drive the real `claude-local` harness against vLLM; self-skip when claude-local is disabled/unreachable. Kept out of `npm test` (needs a running endpoint).
- `npm run chat-log-report` — offline chat-render-log diagnostics (see chat render log below).
- Requires `config.json` (run `./config-gen.sh`); the agent CLIs (`claude`, `pi`) must be on `PATH`.
- Server is **never compiled** — `npm start` runs `tsx server/index.ts` directly. Only the frontend is built (`dist/web`).

## Deployment (systemd user service)

- `./install-service.sh` — one-time: writes/enables `~/.config/systemd/user/agent-remote.service` (runs `npm start`), builds `dist/web` if missing, enables linger. User service (not system): agent CLIs live in per-user paths.
- `./start.sh` / `./stop.sh` / `./restart.sh` wrap `systemctl --user` as the canonical prod runner. `./restart.sh` rebuilds frontend then restarts.
- `./start.sh dev` is the exception: runs the `tsx watch` HMR server directly (pidfile + `setsid`, not systemd). `./stop.sh` prioritizes that pidfile if present, else stops the service.
- `./rebuild.sh` builds frontend only. Frontend change → rebuild + restart; server-only change → plain `systemctl --user restart agent-remote` (no build needed).

## Layout

- `shared/protocol.ts` — WS message + session types, shared by both sides. Harness-agnostic. Also `CommandListing` (the `/api/commands` response shape), `SessionEvent` (command-start/end + cwd events), `FileEntry`/`DirListing`/`FileContent` (the file-editor `/api/files` + `/api/file` shapes), and the normalized chat schema (`SessionUi`, `ChatState`/`ChatMessage`/`ChatPart`, `ChatEvent`, `ChatAction`, `ChatUiRequest`).
- `shared/chat.ts` — `applyChatEvent` reducer folding `ChatEvent`s into `ChatState`; run identically by server (replayable per-session state) and client (live updates). Caps: 200 messages, 20k/tool output, 20 notices.
- `shared/render.ts` — single source of truth for turning a `ChatPart`/`ChatMessage` into displayed HTML (`renderPart`/`renderMessage`, plus the `md`/`escapeHtml`/`argsPreview`/`toolGlyph`/`truncate` primitives). `ChatView.tsx` imports the primitives so the UI can't drift from what's logged; `server/chatLog.ts` calls `renderMessage` to capture what's shown. Imports `marked` — keep it imported only by `ChatView` on the web side so it stays in the lazy chunk.
  - **Tool-call display** is driven by `toolView(part)` (field-driven, harness-agnostic — covers claude's `Edit`/`Read`/`Write`/`Bash` and pi's lowercase `read`/`write`/`bash` by looking at `file_path`/`path`, `command`, `old_string`/`new_string`, `content`, not tool names): returns a `{primary, secondary?, body}` display model. Summary shows a legible subject (`shortenPath` → `…/dir/file.ts`, or the command) not raw JSON; body is a red/green line diff for edits (`lineDiff`, minimal LCS), a code block for Write/Bash content, path+range for Read, or pretty JSON for unknown tools. Both `renderPart` (HTML) and `ChatView`'s `ToolPart` (JSX) consume `toolView` so they can't drift. Tool bodies use `white-space: pre` + horizontal scroll (mobile pans, never wraps per-column). Empty `thinking` parts (claude never streams reasoning) render as a transient pulsing "Thinking…" label and are stripped by the `shared/chat.ts` reducer (`dropEmptyThinking`) once the next part starts or the turn ends.
- `server/` — Node backend. `config.ts` (loads `config.json`), `auth.ts` (password/cookie auth — see below), `llm.ts` (optional LLM assist — see below), `adapters/` (harness boundary, incl. `shell-integration/` injected rc scripts), `sessions/manager.ts` (PTY/streaming/lifecycle + session-event fan-out, harness-agnostic), `commands.ts` (lists cwd executables + `$PATH` + shell aliases + recent/frequent commands from the DB command log; PATH/alias cached 30s), `files.ts` (file-editor backend: list subfolders / read / write, every path confined to a folder root), `chatLog.ts` (chat render log capture — see below), `db.ts` (SQLite: folders, users, auth sessions, command log, resumable chat sessions, chat render log), `index.ts` (HTTP + WS + dev Vite).
- `web/` — React app. `client.ts` (WS client, per-session output buffering + per-session `ChatState`, sticky Ctrl modifier applied in `input()`), `auth.ts` (client-side auth helpers), `Login.tsx` (login/signup gate), `TerminalView.tsx` (xterm per session), `ChatView.tsx` (chat-bubble view for `ui:"chat"` sessions, lazy-loaded, bundles `marked`), `FileEditor.tsx` (the file-editor tab: file picker + CodeMirror, lazy-loaded), `App.tsx` (workspace + mobile keyboard key-bar + editor-tab bookkeeping + AI-assistant mode toggle), `CommandBuilder.tsx` (the `./` command-builder dialog), `commandCatalog.ts` (static well-known-command subcommand/flag data + curated common-command list), `styles.css`.
- **Chat sessions (pi + claude):** `SessionInfo.ui` is `"terminal" | "chat"`, declared by the adapter. Two adapter shapes drive chat, both harness-confined; the manager just folds their `ChatEvent`s into per-session `ChatState`:
  - **`createChatTranslator()` (pi)** — manager spawns the agent **piped, no PTY** (`child_process.spawn`); the `ChatTranslator` maps wire format ↔ schema (`push(stdout) → ChatEvent[]`, `encode(ChatAction) → stdin bytes + synthetic events`). pi runs `pi --mode rpc` (JSONL, manual `\n` framing, never readline — U+2028/29). ALL pi vocabulary in `adapters/pi.ts`.
  - **`createChatSession()` (claude)** — the adapter owns the process/SDK and exposes `start(handlers)`/`action(ChatAction)`/`close()`; the manager drives that interface (no child/translator). `adapters/claude.ts` wraps the **Claude Agent SDK** (`@anthropic-ai/claude-agent-sdk`, `query()` with a pushable async-generator prompt stream). This is what buys **feature parity with the TUI**: `canUseTool` → permission cards (`ui-request` select Allow/Deny, plus "Always allow" when the SDK supplies `suggestions` — returned as `updatedPermissions` so matching calls stop prompting for the session; resolved by `ui-response`). The card carries the tool as `ChatUiRequest.tool` (`{name, args}`), NOT a pre-summarized `message` string, so `ChatView` renders it through the shared `toolView` (same diff/code/path view as the transcript, expanded) instead of raw arg JSON. `onUserDialog`+`supportedDialogKinds:["permission_ask_user_question"]` → the **AskUserQuestion** tool as a structured `ui-request` of `kind:"questions"` (1–4 questions, each 2–4 label+description options, `multiSelect`) instead of a raw permission card; the answers ride back as a PermissionResult `updatedInput.answers` (question text → chosen label(s), multi joined `", "`), mirroring the TUI. Other dialog kinds aren't declared → CLI applies its default, `setModel`/`supportedModels` → model switcher, `setPermissionMode` → mode toggle (a fixed `modes` event lists Default/Plan/Allow-all-edits/Auto; `set-mode` action → `setPermissionMode`), `supportedCommands` → slash-command palette (all fetched eagerly on start, before the first prompt). SDK message mapping mirrors the raw stream-json shapes (stream_event deltas → part events; `input_json_delta` accumulates tool args, emitted at `content_block_stop`; `user` tool_result → `tool-end`; `result` → busy false). **Compaction:** the SDK `system`/`compact_boundary` message (manual `/compact` or auto when the window fills) → an info `notice` (`Compacted context (pre → post tokens) [manual|auto]`) so the transcript marks the boundary instead of silently swapping history for a summary; a `system`/`status` with `compact_result:"failed"` → an error notice. `permissionMode: "default"` = auto-allow read-only, prompt (via `canUseTool`) for the rest. Abort = `query.interrupt()`.
  - **Billing:** claude stays flat-rate subscription — the SDK drives the Claude Code CLI on the user's OAuth creds. Never set `ANTHROPIC_API_KEY` or `--bare` (forces metered API) **on the production `claude` harness**. Verified on subscription.
  - **`claude-local` (token-free e2e testing):** an optional second instance of the *same* Claude adapter (`config.harnesses.claudeLocal`, id `claude-local`, built in `registry.ts`), whose `HarnessConfig.env` points the Claude CLI at a local Anthropic-compatible endpoint. vLLM natively serves `/v1/messages`, so `env` sets `ANTHROPIC_BASE_URL=http://localhost:8000` + `ANTHROPIC_API_KEY/AUTH_TOKEN=dummy` + `ANTHROPIC_DEFAULT_{OPUS,SONNET,HAIKU}_MODEL=<served model>` (`CLAUDE_CODE_DISABLE_NONESSENTIAL_TRAFFIC=1` to stop catalog pings). `createClaudeAdapter(cfg, {id,name})` takes the id/name override; `env` threads to `ClaudeChatSession` → `query({env})`. **Gotcha:** the SDK `env` option *replaces* the subprocess env, so the adapter spreads `process.env` first (keeps PATH/HOME). This exercises the whole real stack (SDK message mapping, permission cards, streaming, resume, WS/ChatState/ChatView) with zero Claude tokens — the dummy key is scoped to this harness's env only, never the production `claude` path. vLLM model must have tool-calling enabled (`--enable-auto-tool-choice --tool-call-parser …`) and ≥32K context. `scripts/claude-local-smoke.ts` drives one turn through the adapter as a quick check. Automated coverage: `server/adapters/*.e2e.test.ts` (run via `npm run test:e2e`, shared helpers in `claude-local.testkit.ts` — `claudeLocal()`/`endpointUp()`/`ChatDriver`) — `claude.e2e.test.ts` drives a real edit turn and asserts the backend asks permission (Edit → `ui-request`) and parses the streamed tool args (`input_json_delta` → well-formed Edit object); `claude-resume.e2e.test.ts` plants a codeword, resumes by key, and asserts the prior turn is replayed from disk AND the model recalls the codeword (context restored, not a fresh session); `claude-compact.e2e.test.ts` drives `/compact` and asserts the backend surfaces the compaction boundary as a notice (deterministic) plus a softer check that the model still recalls the pre-compaction codeword from the summary. All fold through the shared reducer/renderer and self-skip unless claude-local is enabled and its endpoint answers. Not a substitute for an occasional real-token smoke of OAuth/model-switcher behavior.
  - Manager mirrors busy→`currentCommand` (pulsing dot unchanged; the `index.ts` command recorder skips `ui==="chat"` so prompts stay out of `./`-builder recents). On WS connect the server sends one `chatState` snapshot per chat session (scrollback replay); live `chatEvent`s follow. `ChatView.tsx` renders bubbles (markdown via `marked`, raw HTML escaped), collapsible thinking/tool parts, permission/UI cards (confirm/select/input), a model `<select>` + permission-mode `<select>` + slash-command palette (shown when `ChatState.models`/`.modes`/`.commands` are non-empty), queued prompts, and a composer (Enter sends; Send↔Steer while busy + Stop=abort). Terminal-only UI (terminal key-bar, select mode, builder) gated off via `activeIsChat`. ChatView has its **own** mobile key-bar (`.chat-key-bar`, gated on the `keyboardOpen` prop from `App.tsx`'s `keyboard.open`) above the composer with keys the soft keyboard lacks for prompt-writing — currently backtick + newline — inserting at the textarea caret via `insertAtCursor` (reuses `.key-button`). Manually running `pi`/`claude` inside a Terminal session stays raw PTY. New chat harness = adapter with `createChatTranslator` *or* `createChatSession`; no UI/manager changes.
- **Resume (chat sessions):** harness-agnostic via an opaque "resume key". `SessionOptions.resume` (input) → adapter restores prior state; `ChatSessionHandlers.onResumable(key)` (output) → manager persists it. Only the **claude** adapter implements resume so far: passes `resume` to `query()` (restores the model's context) and reports the SDK `session_id` from the `system`/`init` message. **Crucial gotcha:** the SDK does NOT stream the prior conversation back on resume (nothing is emitted until the first new prompt, and even then only the new turn — no `isReplay` messages). So the adapter rebuilds the visible transcript itself in `replayHistory()` by reading the on-disk JSONL via the SDK's `getSessionMessages(sessionId, { dir: cwd })`, folding those whole messages through the same `replayAssistantMessage`/`replayUserMessage`/`handleToolResults` helpers (same ChatEvent vocabulary). The `streamedThisTurn` flag (set on stream `message_start`, cleared on `result`) still guards live delta-built turns against the SDK's redundant whole-message echo; the `isReplay` branch in `handleMessage` is now a vestigial safety net (the SDK doesn't send those). `db.ts` `chat_sessions` table (keyed by resume key + folder, title = first user prompt) survives restarts; `index.ts`'s server-global subscriber persists on `onResumable`/first `user-message`. `GET/DELETE /api/resumable?cwd=|key=` (auth'd + folder-allowlisted) list/forget; the list hides currently-live keys (`manager.liveResumeKeys()`). UI: resume is triggered from **`/resume` in `ChatView`'s slash-command menu** (mirrors the CLI), shown when the folder has resumable sessions. `/resume` is a **client-only** menu entry (not a harness `state.commands` command): `ChatView` takes `canResume`/`onResume` props; picking it calls `onResume` (not `insertCommand`). The dialog state itself stays folder-level in `Workspace` (`resumable`/`resumeDialogOpen`), so `onResume={() => setResumeDialogOpen(true)}`. It opens a modal dialog listing the folder's resumable sessions; picking one calls `client.start(harnessId, folder, resumeKey)` and closes the dialog, `×` forgets one. Reachable only when a chat tab is active (no standalone/sidebar entry). pi/terminal ignore `resume`.
- **Chat render log (diagnostics):** to review/improve how chat messages are displayed, `server/chatLog.ts` persists, per finalized chat message, both the original normalized `ChatMessage` and its rendered form (`shared/render.ts`'s `renderMessage`: per-part component name + CSS class + produced HTML). Captured harness-agnostically in `index.ts`'s server-global `manager.subscribe` (`onChatEvent`, skipping the high-frequency `part-delta`/`tool-update` — final form lands on assistant-end/tool-end); deduped by a per-message JSON signature before any rendering (cheap to call often), cleared on `onRemoved`. `db.ts` `chat_render_log` table keyed by `(session_id, message_id)`, upserted so late tool results refresh the row, capped ~20k rows (probabilistic prune). Read-only review via `GET /api/chat-log?session=&limit=` (auth'd; returns rows with `original`/`rendered` parsed from JSON). No UI/client involvement — pure server-side diagnostics. **`npm run chat-log-report`** (`scripts/chat-log-report.ts`) is the offline CLI over the same table: summary (part/component/tool/status tallies) + heuristic issue flags; modes `--dump=<msgId>` (full original+rendered JSON), `--issue=<name>`, `--tool=<name>`, `--part=<type>`; filters `--harness= --session= --limit= --samples=`. Add a heuristic = one entry in its `ISSUES` map. Use it instead of ad-hoc SQLite scripts.
- **File editor:** a fourth tab type alongside claude/pi/terminal, added from the same `+` menu ("File edit"). NOT a harness — it's a **client-only tab** tracked in `App.tsx` (`editors` state), never by the PTY session manager; it's backed by REST, not `/ws`. Two steps in `FileEditor.tsx`: a lazy folder tree (file picker) then a CodeMirror 6 editor with a Save button (Ctrl/Cmd+S). Reads/writes via `GET /api/files?cwd=&path=` (dir listing), `GET/PUT /api/file?cwd=&path=` — all auth'd + folder-allowlisted like `/api/commands`; `server/files.ts` confines every path to the folder root (blocks `..`/absolute escapes), refuses binary/oversized files. CodeMirror + language packs are code-split (dynamic `import()` behind `React.lazy`) so they load only when an editor tab is opened. Save creates new files too (via the picker's "New file"). Terminal-only UI (key-bar, command builder) is gated off when the active tab is an editor (`activeIsEditor`).
- **Command builder:** `./` key-bar button opens `CommandBuilder`. Lists recent/frequent (from the DB command log), cwd executables, then aliases/common/full-`$PATH` (server `GET /api/commands?cwd=` — auth'd, cwd must be a known folder). Picking a plain command then filters static subcommand/flag suggestions from `commandCatalog.ts` (free-text for unknown commands); picking a recent/frequent entry (a whole command line) inserts it directly. **Inserts** the assembled line into the PTY without a trailing newline — user reviews and presses Enter.
- **Recent/frequent commands:** sourced from the DB `commands` log (every command run in a Terminal session, with its cwd — see shell integration), not shell-history files. `recentCommands`/`frequentCommands` in `db.ts` rank distinct commands with the requested cwd preferred (same-cwd first, then global). Still filtered per-cwd in `listCommands`: only relative-path invocations (`./x`) are cwd-dependent and dropped if absent from the folder; plain names and absolute paths always kept. Ranked pool is larger than shown so the filter can still fill `HISTORY_LIMIT`. Cold-start: empty until commands are run through a Terminal session.
- **Shell integration / session events:** the `terminal` adapter (`adapters/shell.ts`) detects zsh/bash from the configured command and injects a startup script (zsh via hijacked `ZDOTDIR`+`USER_ZDOTDIR` → `shell-integration/zdotdir/`; bash via `--rcfile shell-integration/bash-rc.sh`) that still sources the user's own rc, then emits VS Code **OSC 633** markers (`E`=command line, `C`=exec, `D;code`=done, `P;Cwd=`=cwd). The adapter's `createEventParser()` strips those markers from the PTY stream and yields `SessionEvent`s; `manager.ts` updates `SessionInfo.cwd` and `SessionInfo.currentCommand` (set on `command-start`, cleared on `command-end`; shown in the UI session list with a pulsing dot) live and fans events to listeners (`onEvent`). Other harnesses (claude/pi) and non-zsh/bash shells get no parser — they stream raw, unchanged. Adding event support to a harness = give its adapter a `createEventParser`; keep the OSC knowledge in the adapter. Command logging lives in a single server-global `manager.subscribe` in `index.ts` (NOT the per-connection one, or every browser double-records).
- **Dynamic arg resolvers:** a catalog `ArgNode.source` names a server-side resolver (`RESOLVERS` in `commands.ts`) that runs a *fixed* command and returns live suggestions (e.g. `docker logs` → container names from `docker ps -a`; `git checkout` → branches; `npm run` → package.json scripts). Client calls `GET /api/resolve?id=&cwd=` with only the resolver **id** (never a command) — auth'd + folder-allowlisted; results cached 3s; failures degrade to free-text. Add a resolver = add an entry to `RESOLVERS` + set `source` on the catalog node.
- **Mobile keyboard:** `App.tsx` sizes `.app` to `visualViewport.height` when the keyboard is up and floats the key-bar above it; `index.html` sets `interactive-widget=resizes-content`. Together these stop the page being panned under the keyboard — don't revert without re-checking that.
- **Authentication:** the whole app is login-gated. `server/auth.ts` owns all of it — scrypt password hashing (`salt:hash`), server-side session tokens stored in SQLite (`users`/`auth_sessions` in `db.ts`) and set as an **HttpOnly cookie** (never exposed to page JS), plus the HTTP auth routes. The rest of the server only asks "who is this request?" via `authedUser()`; every `/api` route and the `/ws` upgrade are gated on it. `web/Login.tsx` is the login/signup gate and `web/auth.ts` the client helpers. Keep auth concerns in `auth.ts` — the rest of the code stays auth-agnostic.
- **LLM assist / AI-assistant mode (optional):** `server/llm.ts` talks to an **OpenAI-compatible endpoint** (`config.json` `llm` block; default local vLLM at `http://localhost:8000/v1`) to auto-answer chat `ui-request`s on the user's behalf — permission prompts (`kind:"select"`/`"confirm"` → allow/deny with a terse reason) and AskUserQuestion dialogs (`kind:"questions"` → pick option labels). Everything is **best-effort/fail-safe**: it polls the endpoint's `/models` for health, and any error/unreachable endpoint degrades to "unavailable" so the UI falls back to the normal manual flow. Harness-agnostic — lives outside adapters, driven from `index.ts`; protocol types (`LlmStatus`/`LlmEvaluateRequest`/`LlmDecision`) in `shared/protocol.ts`; the mode is toggled per-user in `web/App.tsx` and gated on `llmStatus().available`. The conservative default (no user instructions) only allows trivially-safe read-only operations. This is the one **built** LLM feature; the rest below remain unbuilt.

## What this is

- Harness-agnostic web remote for AI coding agents.
- Single webpage to manage local agent sessions: Claude Code + pi first, other CLI harnesses later.
- User-facing overview: `README.md`.

## Architecture (settled)

- **Frontend:** React + Vite. Renders sessions, streams output, sends input back.
- **Backend:** Node. Spawns each agent as a **local subprocess in a PTY** (`node-pty`), supervises lifecycle, streams raw terminal I/O to the browser via WebSocket.
- **Terminal:** browser renders sessions with **xterm.js**; keystrokes/resize go back over `/ws`. One persistent xterm per session (hidden when inactive) so scrollback survives tab switches.
- **Single port:** UI, `/api`, and `/ws` all listen on one port (default 4000). Dev embeds Vite in **middleware mode** in the Node server (`--dev` flag, not env); prod serves `dist/web`. `/ws` and Vite HMR coexist via `noServer` + manual `upgrade` routing.
- **Harness adapters:** core abstraction. An adapter describes *how to invoke* an agent (`HarnessInvocation`: command/args/env) plus optionally how to translate its protocol (`createEventParser` for PTY event markers, `createChatTranslator` for chat sessions); all process/streaming/lifecycle is harness-agnostic in `sessions/manager.ts`.
- **Hard constraint:** keep harness-specific logic confined to adapters. UI/session-mgmt/LLM features stay harness-agnostic. New agent = new adapter, never UI/core changes. Leakage into shared code = design smell.

## Config

- No env variables. All config lives in `config.json` (gitignored).
- `config-gen.sh` runs a setup dialog to generate `config.json`. Shape in `config.example.json`.
- Keys: `harnesses` (per-harness `enabled`/`command`, optional `env` for the process — used by `claudeLocal`), `llm` (`provider`/`baseUrl`/`model` for the optional LLM assist — see above), `server.port`.
- Never reintroduce `process.env`-style config or `.env` files.

## LLM features

- **Built:** AI-assistant mode — auto-answering chat permission/question dialogs via an OpenAI-compatible endpoint (`server/llm.ts`; see the LLM-assist bullet in Layout).
- **Planned (not built):** session summaries, natural-language task routing, notification/triage.
- All layered on the harness-agnostic core; consume the common session interface; must not depend on a specific harness.
